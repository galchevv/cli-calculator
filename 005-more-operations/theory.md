# Модуль 5. Расширение операций — теория

## Контекст

Твой `main` после модуля 4 уже довольно длинный — там и проверка аргументов, и парсинг чисел, и `switch` с операциями, и обработка ошибок. Если сейчас добавить ещё несколько операций (степень, корень, остаток), `main` превратится в монстра на 100+ строк, в котором сложно ориентироваться.

В этом модуле ты научишься **раскладывать код на функции**. Это самый базовый инструмент борьбы со сложностью в любом языке программирования.

## Ядро: функция как единица смысла

Функция в Go выглядит так:

```go
func add(a, b float64) float64 {
    return a + b
}
```

Разберём по частям:

- `func` — ключевое слово, объявление функции
- `add` — имя функции
- `(a, b float64)` — параметры. Тут два параметра одного типа, и есть короткая запись: вместо `a float64, b float64` можно `a, b float64`
- `float64` после скобок — **тип возвращаемого значения**
- `return ...` — что вернуть

Использовать так:

```go
result := add(2, 3)  // result == 5
```

## Почему функции — не просто «способ переиспользовать код»

Многие думают про функции так: «вынесли повторяющийся кусок, чтобы не дублировать». Это правда, но это не главное.

Главное — функция **даёт имя куску логики**. Когда ты читаешь `result := add(a, b)`, ты сразу понимаешь, что здесь происходит. Когда ты читаешь `result := a + b` среди 50 строк другого кода — тебе надо включать голову. Имя функции — это **сжатая документация**.

Это особенно важно, когда кусок кода выполняет что-то более сложное. Сравни:

```go
// Без функций — надо вчитываться
if op == "/" {
    if b == 0 {
        fmt.Fprintln(os.Stderr, "Ошибка: деление на ноль")
        os.Exit(1)
    }
    result = a / b
}

// С функцией — намерение видно сразу
result, err := divide(a, b)
if err != nil { ... }
```

Имя `divide` сразу говорит, что мы делим. Логика «как именно делим, что проверяем» — спрятана внутри. Это называется **инкапсуляция**.

## Возврат нескольких значений

Ты уже видел это в `strconv.ParseFloat` — функция вернула число и ошибку. Это норма Go: функция, которая может провалиться, возвращает **(результат, ошибка)**:

```go
func divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, fmt.Errorf("деление на ноль")
    }
    return a / b, nil
}
```

Несколько важных моментов:

1. Тип возвращаемого значения теперь в скобках: `(float64, error)`. Скобки обязательны при двух и более возвращаемых типах.
2. В случае ошибки результат — нулевое значение (`0`). Вызывающая сторона не должна его использовать.
3. В нормальном случае ошибка — `nil`.
4. `errors.New` и `fmt.Errorf` — оба создают ошибку. Используй `Errorf`, когда нужно подставить значения.

## Принцип единственной ответственности

Хорошая функция делает **одну вещь** и делает её хорошо. Плохая функция — это «бог-функция», которая и парсит, и считает, и выводит, и пишет в файл.

Для калькулятора естественное разбиение такое:

| Функция | Ответственность |
|---------|------------------|
| `parseArgs` | Прочитать аргументы, провалидировать, вернуть числа и операцию |
| `add`, `subtract`, `multiply`, `divide` | По одной арифметической операции |
| `calculate` | Координатор: принимает числа и операцию, возвращает результат |
| `main` | Точка входа: вызывает остальное и обрабатывает ошибки |

Каждая функция знает только своё. `add` не знает, откуда взялись числа. `parseArgs` не знает, что с ними будут делать. `calculate` не знает, как именно складывать. Это **разделение ответственностей**, и оно делает код понятным.

## Пример с разбором

```go
package main

import (
    "fmt"
    "os"
    "strconv"
)

func add(a, b float64) float64      { return a + b }
func subtract(a, b float64) float64 { return a - b }
func multiply(a, b float64) float64 { return a * b }

func divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, fmt.Errorf("деление на ноль")
    }
    return a / b, nil
}

func calculate(a, b float64, op string) (float64, error) {
    switch op {
    case "+":
        return add(a, b), nil
    case "-":
        return subtract(a, b), nil
    case "*":
        return multiply(a, b), nil
    case "/":
        return divide(a, b)
    default:
        return 0, fmt.Errorf("неизвестная операция: %s", op)
    }
}

func parseArgs(args []string) (float64, string, float64, error) {
    if len(args) != 4 {
        return 0, "", 0, fmt.Errorf("ожидаю 3 аргумента, получил %d", len(args)-1)
    }

    a, err := strconv.ParseFloat(args[1], 64)
    if err != nil {
        return 0, "", 0, fmt.Errorf("первый аргумент не число: %s", args[1])
    }

    b, err := strconv.ParseFloat(args[3], 64)
    if err != nil {
        return 0, "", 0, fmt.Errorf("третий аргумент не число: %s", args[3])
    }

    return a, args[2], b, nil
}

func main() {
    a, op, b, err := parseArgs(os.Args)
    if err != nil {
        fmt.Fprintln(os.Stderr, "Ошибка:", err)
        os.Exit(1)
    }

    result, err := calculate(a, b, op)
    if err != nil {
        fmt.Fprintln(os.Stderr, "Ошибка:", err)
        os.Exit(1)
    }

    fmt.Println("Результат:", result)
}
```

**Что важно в этой структуре:**

- `main` стал коротким и читаемым. Он только координирует.
- Каждая операция — отдельная функция, которую можно тестировать в изоляции.
- `parseArgs` отделяет «откуда пришли данные» от «что мы с ними делаем».
- Ошибки бегут «снизу вверх» — каждая функция возвращает их вверх, `main` обрабатывает.

## Однострочные функции — нормально?

Заметь, что `add`, `subtract`, `multiply` — это функции из одной строки. Кажется, что это излишнее. Так ли это?

Аргументы за:

- **Унификация.** Все четыре операции имеют одинаковый «фасад». Если завтра `add` усложнится (например, добавить логирование) — это будет одно изменение в одном месте.
- **Тестируемость.** Каждую функцию можно проверить отдельным тестом.
- **Имя как документация.** `add(a, b)` чуть-чуть, но яснее, чем `a + b` — особенно когда `a` и `b` имеют сложные имена.

Аргументы против:

- Лишний код. Для калькулятора, возможно, перебор.

В реальном проекте ты будешь принимать решение каждый раз заново. Для **обучения** сейчас лучше сделать с функциями — это закрепит навык декомпозиции.
