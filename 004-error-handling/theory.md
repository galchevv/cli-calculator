# Модуль 4. Обработка ошибок — теория

## Контекст

В предыдущих модулях ты накопил несколько «дыр» в программе:

- Если ввести букву вместо числа — программа выводит `0` и делает вид, что всё хорошо
- Если разделить на ноль — для `float64` получаем `+Inf` или `NaN`
- Если ввести неизвестную операцию — `switch` ничего не сделает, программа выведет `0`

Все эти случаи объединяет одно: **программа не упала, но и не сделала то, что от неё ожидали**. Это называется **тихий баг** — самый коварный вид ошибок. Программа врёт пользователю и делает вид, что всё хорошо.

В этом модуле ты научишься их ловить и реагировать **явно**.

## Ядро: ошибка как значение

Это, возможно, самая важная концепция Go в принципе. Запомни её на всю жизнь.

В большинстве популярных языков (Java, Python, C#, JavaScript) ошибки — это **исключения**. Когда что-то идёт не так, программа «выбрасывает» исключение, которое летит вверх по стеку, пока кто-то не поймает его в `try/catch`.

В Go **исключений нет**. Вместо этого функции, которые могут потерпеть неудачу, **возвращают ошибку как обычное значение** — рядом с результатом.

Идиома выглядит так:

```go
result, err := какаяТоФункция()
if err != nil {
    // что-то пошло не так
    // обрабатываем ошибку
}
// если дошли сюда — всё хорошо, можно использовать result
```

## Почему так

Это решение Go выглядит непривычно, но за ним стоит сильная философия:

1. **Ошибки видны в коде.** В языках с исключениями ты не знаешь, какая функция может «взорваться». В Go — знаешь сразу: если функция возвращает `(result, error)`, она может провалиться. Это написано в её сигнатуре.
2. **Ошибки нельзя случайно проигнорировать.** Точнее, можно — но это требует **явного** действия (написать `_`). Если ты просто принял два значения через `result, err := ...` и не проверил `err` — линтер на тебя наорёт.
3. **Поток управления линеен.** Нет невидимых прыжков в `catch` за тысячу строк отсюда. Ошибка обрабатывается **здесь же, на следующей строке**. Код легче читать и отлаживать.

Минус — кода становится больше. Ты увидишь, что половина любого Go-проекта — это `if err != nil { return err }`. К этому привыкаешь быстро, и взамен получаешь предсказуемость, которая в продакшене бесценна.

## Структура: тип error

`error` в Go — это **встроенный тип** (точнее, интерфейс, но это пока неважно). Значение типа `error` может быть либо `nil` (значит, всё хорошо), либо реальной ошибкой с сообщением.

```go
var err error  // по умолчанию nil

err = errors.New("что-то сломалось")  // теперь не nil
```

Создать ошибку можно двумя способами:

```go
import "errors"
import "fmt"

err1 := errors.New("деление на ноль")

// fmt.Errorf — как Println, но для ошибок
err2 := fmt.Errorf("неверная операция: %s", op)
```

`fmt.Errorf` особенно полезен — он умеет подставлять значения в текст ошибки. `%s` означает «вставить сюда строку».

## Идиома: ранний возврат

Один из самых частых паттернов в Go — **early return**. Если что-то пошло не так, **сразу выходим из функции**, не углубляясь в нормальный путь.

```go
func parseNumber(s string) (float64, error) {
    n, err := strconv.ParseFloat(s, 64)
    if err != nil {
        return 0, fmt.Errorf("не могу превратить '%s' в число", s)
    }
    return n, nil
}
```

Заметь несколько вещей:

1. Если есть ошибка — возвращаем её сразу. Не пытаемся «всё равно что-то сделать».
2. В случае ошибки результат — **нулевое значение** (`0` для `float64`, `""` для строки, `nil` для указателя). Вызывающая сторона не должна его использовать.
3. В нормальном случае возвращаем результат и `nil` как ошибку.

## Обработка в main

В функции `main` ты обычно не «возвращаешь» ошибку дальше — некуда. Вместо этого ты её **выводишь** и завершаешь программу.

```go
func main() {
    if len(os.Args) < 4 {
        fmt.Fprintln(os.Stderr, "Использование: calc <число> <операция> <число>")
        os.Exit(1)
    }

    a, err := strconv.ParseFloat(os.Args[1], 64)
    if err != nil {
        fmt.Fprintln(os.Stderr, "Первый аргумент должен быть числом:", err)
        os.Exit(1)
    }

    // ... и так далее
}
```

Здесь две новые штуки:

- **`fmt.Fprintln(os.Stderr, ...)`** — это `Println`, но в поток ошибок (stderr), а не в обычный вывод (stdout). Зачем? Чтобы можно было перенаправлять вывод и ошибки отдельно. Это правильная Unix-привычка.
- **`os.Exit(1)`** — завершение с кодом ошибки. Скрипты, которые запускают твою программу, увидят этот код и поймут, что что-то не так.

## Деление на ноль

Для `float64` Go не выбрасывает ошибку при делении на ноль — ты получишь `+Inf`. Это иногда полезно (в численных расчётах), но для калькулятора неуместно. Ноль нужно поймать **до** деления:

```go
if op == "/" && b == 0 {
    fmt.Fprintln(os.Stderr, "Ошибка: деление на ноль")
    os.Exit(1)
}
```

## Полный пример с разбором

```go
package main

import (
    "fmt"
    "os"
    "strconv"
)

func main() {
    if len(os.Args) != 4 {
        fmt.Fprintln(os.Stderr, "Использование: calc <число> <операция> <число>")
        os.Exit(1)
    }

    a, err := strconv.ParseFloat(os.Args[1], 64)
    if err != nil {
        fmt.Fprintf(os.Stderr, "Первый аргумент не число: %s\n", os.Args[1])
        os.Exit(1)
    }

    op := os.Args[2]

    b, err := strconv.ParseFloat(os.Args[3], 64)
    if err != nil {
        fmt.Fprintf(os.Stderr, "Третий аргумент не число: %s\n", os.Args[3])
        os.Exit(1)
    }

    var result float64
    switch op {
    case "+":
        result = a + b
    case "-":
        result = a - b
    case "*":
        result = a * b
    case "/":
        if b == 0 {
            fmt.Fprintln(os.Stderr, "Ошибка: деление на ноль")
            os.Exit(1)
        }
        result = a / b
    default:
        fmt.Fprintf(os.Stderr, "Неизвестная операция: %s\n", op)
        os.Exit(1)
    }

    fmt.Println("Результат:", result)
}
```

**Что здесь важно:**

- Каждая операция, которая может провалиться, проверяется **сразу же**.
- В каждой ветке ошибки — понятное сообщение, которое говорит, **что именно** не так.
- Все ошибки идут в `stderr`, результат — в `stdout`.
- Нормальный путь (последняя строка) не загромождён проверками — все они уже выше.
